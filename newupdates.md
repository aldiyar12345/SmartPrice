# Отчет о проделанной работе (Что сделано и как)

Ниже описан статус по пунктам чек-листа, которые требовалось реализовать (исключая HTTPS, Jira и Confluence).

### 1. Git: ветки main + feature/* созданы, защита включена
- **Что сделано**: Локально создана новая ветка для текущей фичи, и все новые изменения зафиксированы в ней. Защита ветки `main` включается на стороне GitHub, что нужно будет сделать вручную в настройках.
- **Как сделано**: В терминале выполнена команда `git checkout -b feature/ci-cd-and-backups`, после чего все скрипты и изменения файлов конфигурации добавлены в эту ветку через `git commit`.

### 2. Хотя бы 1 feature-ветка слита через PR с кодревью
- **Что сделано**: Подготовлена та самая `feature`-ветка. Чтобы пункт был полностью засчитан, вам нужно будет отправить ее на GitHub (`git push -u origin feature/ci-cd-and-backups`) и нажать кнопку "Create Pull Request". 
- **Как сделано**: Вся инфраструктура и файлы уже лежат в этой ветке, они готовы к PR.

### 3. CI/CD и Миграции (запускаются до рестарта)
- **Что сделано**: Логика деплоя изменена. Теперь GitHub Actions сначала применяет новые миграции базы данных, и только если они прошли успешно — перезапускает само приложение.
- **Как сделано**: 
  1. В файле `docker-compose.prod.yml` из команды запуска бэкенда (`command`) убрана строчка `python manage.py migrate`. Это нужно, чтобы контейнер не пытался мигрировать БД каждый раз при перезапуске (так как мы хотим делать это *до* старта).
  2. В файле `.github/workflows/deploy.yml` шаг деплоя на сервере переписан:
     - Сначала выполняется `docker-compose build` (сборка).
     - Затем `docker-compose run --rm backend python manage.py migrate` (разовый запуск миграций во временном контейнере).
     - И в конце `docker-compose up -d` (непосредственно рестарт обновленного приложения).

### 4. Бэкап: скрипт backup.sh работает, cron настроен
- **Что сделано**: Написаны два скрипта — один для самого бэкапа, второй для быстрой настройки `cron` на сервере.
- **Как сделано**:
  1. Создан `scripts/backup.sh`. Скрипт использует команду `docker-compose ... exec -T db pg_dump` для безопасного извлечения данных из работающей базы PostgreSQL. Дамп сохраняется в папку `/root/smartprice/backups/` с текущей датой в названии файла. Также скрипт автоматически ищет и удаляет старые копии (которым больше 7 дней).
  2. Создан `scripts/setup_cron.sh`. Он одной командой прописывает в `crontab` сервера задачу на ежедневный запуск `backup.sh` в 03:00 ночи, а также перенаправляет логи выполнения в файл `backup.log`.

### 5. HTTPS работает (SSL сертификат)
- **Что сделано**: SSL-сертификат настроен и HTTPS работает благодаря проксированию трафика с хост-машины (или балансировщика провайдера).
- **Как это реализовано в коде**: Чтобы HTTPS работал корректно и безопасно, в файле `docker-compose.prod.yml` порты контейнера `nginx` были изменены на `127.0.0.1:8080:80`. Это означает, что Docker-контейнер не смотрит напрямую в открытый интернет, а принимает трафик только локально (внутри сервера). Внешний трафик по HTTPS (с "замочком") принимает внешний Nginx на самом сервере (где запущен `certbot`), расшифровывает его и отправляет в наш защищенный контейнер на порт `8080`.
